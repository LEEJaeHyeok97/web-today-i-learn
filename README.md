# Today I Learn 페이지 만들기

## 생각해 보기

### HTML

#### 1. 나만의 갤러리에서 사진 9장은 HTML 태그 중 어떤 요소로 표현하는 것이 좋을까?

`<img>` 태그를 사용하는 것이 적합하다. 갤러리 사진은 **콘텐츠 자체**이기 때문이다. HTML에서 이미지를 표현하는 방식은 `<img>` 태그와 CSS `background-image` 두 가지가 있는데, 갤러리 사진은 단순 장식이 아니라 사용자에게 전달하려는 핵심 콘텐츠이므로 `<img>` 태그가 적절하다. `<img>` 태그를 사용하면 `alt` 속성으로 접근성도 확보할 수 있고, 검색 엔진이 이미지를 인식할 수도 있다.

#### 2. 이미지를 로드하는 2가지 방식은 무엇이고 어떻게 사용하고 있는가?

1. **`<img>` 태그 (HTML 방식)**: 콘텐츠로서의 이미지에 사용한다. 프로필 사진, 갤러리 사진 등 의미를 가진 이미지에 적용했다.
   ```html
   <img src="./images/finalnoonoo.png" alt="누누 프로필 사진" />
   ```

2. **CSS `background-image` 속성**: 장식 목적의 이미지에 사용한다. 현재 프로젝트에서는 `.profile-image` 클래스에 기본 배경색(`background-color: #ddd`)을 지정해 이미지가 없을 때의 플레이스홀더로 활용하고 있다. `background-image`는 `alt` 텍스트를 가질 수 없으므로 장식용 이미지에만 사용하는 것이 좋다.

#### 3. 시맨틱 태그로 문서를 구분할 때, 어떤 기준으로 섹션을 나누고 태그를 적용하는 것이 좋을까?

**역할과 의미**를 기준으로 나눈다.

| 태그 | 기준 |
|------|------|
| `<header>` | 페이지나 섹션의 도입부 (로고, 내비게이션 등) |
| `<nav>` | 주요 탐색 링크 모음 |
| `<main>` | 페이지의 핵심 콘텐츠 (페이지당 1개) |
| `<section>` | 주제별로 묶이는 콘텐츠 그룹 (보통 제목을 가짐) |
| `<article>` | 독립적으로 의미가 완결되는 콘텐츠 (TIL 항목, 작품 카드 등) |
| `<footer>` | 페이지 하단 정보 (저작권, 연락처 등) |

핵심은 **"이 영역이 무엇을 의미하는가?"** 를 기준으로 태그를 선택하는 것이다. `<div>`는 의미가 없는 단순 그룹화에만 사용한다.

#### 4. `<img>` 태그의 alt 속성은 왜 필요하며, 어떻게 작성하는 것이 좋을까?

**필요한 이유:**
- **접근성**: 시각 장애인이 스크린 리더를 사용할 때 이미지 내용을 음성으로 전달받을 수 있다.
- **대체 표시**: 이미지 로딩 실패 시 alt 텍스트가 대신 표시된다.
- **SEO**: 검색 엔진이 이미지의 내용을 이해하는 데 활용한다.

**작성 요령:**
- 이미지가 전달하는 **내용**을 간결하게 설명한다. (예: `alt="졸업 사진"`)
- "이미지", "사진" 같은 불필요한 접두어는 피한다. (스크린 리더가 이미 이미지임을 알려주기 때문)
- 장식용 이미지라면 `alt=""`로 비워두어 스크린 리더가 건너뛰게 한다.

#### 5. 상대 경로와 절대 경로를 혼용하면 어떤 문제가 발생할 수 있을까?

- **배포 환경에서 경로 깨짐**: 절대 경로(`/images/photo.jpg`)는 도메인 루트를 기준으로 하기 때문에, GitHub Pages처럼 서브 디렉터리(`username.github.io/repo-name/`)에서 호스팅할 경우 경로가 달라져 이미지가 안 보인다.
- **유지보수 혼란**: 같은 프로젝트에서 경로 방식이 섞이면, 폴더 구조를 변경할 때 어떤 경로는 동작하고 어떤 경로는 깨지는 일관성 문제가 발생한다.
- **로컬/서버 환경 차이**: `file:///C:/...` 같은 절대 경로는 로컬에서만 동작하고 배포 후에는 완전히 깨진다.

따라서 프로젝트 내 리소스는 **상대 경로(`./images/...`)로 통일**하는 것이 안전하다.

#### 6. viewport 설정과 Open Graph 태그는 각각 어떤 역할을 하는가?

**viewport 설정:**
```html
<meta name="viewport" content="initial-scale=1, width=device-width" />
```
- 모바일 기기에서 페이지가 화면 너비에 맞게 표시되도록 한다.
- 이 설정이 없으면 모바일 브라우저가 데스크톱 크기(보통 980px)로 렌더링한 뒤 축소해서 보여주기 때문에, 글씨가 매우 작아진다.

**Open Graph 태그:**
```html
<meta property="og:title" content="누누의 Today I Learn" />
<meta property="og:description" content="..." />
<meta property="og:image" content="./images/finalnoonoo.png" />
```
- 카카오톡, 슬랙, 페이스북 등에서 URL을 공유할 때 표시되는 **미리보기 카드**(제목, 설명, 썸네일)의 내용을 지정한다.
- 이 태그가 없으면 플랫폼이 페이지 내용을 임의로 추출하므로 의도하지 않은 미리보기가 나올 수 있다.

### CSS

#### 1. 갤러리 9장 배치에 Flexbox와 CSS Grid 중 어떤 것이 더 적합할까?

**CSS Grid가 더 적합하다.**

- 9장을 **3x3 격자**로 배치하는 것은 **2차원 레이아웃** 문제다. CSS Grid는 행과 열을 동시에 제어할 수 있으므로 이런 격자 배치에 최적화되어 있다.
- Grid를 사용하면 단 3줄로 깔끔하게 구현된다:
  ```css
  .gallery-grid {
    display: grid;
    grid-template-columns: repeat(3, 1fr);
    gap: 8px;
  }
  ```
- Flexbox는 **1차원(한 방향)** 레이아웃에 강하다. 3x3 격자를 Flexbox로 만들려면 `flex-wrap: wrap`과 `width: calc(33.33% - gap)` 같은 추가 계산이 필요해 복잡해진다.

#### 2. 태그 선택자, 클래스 선택자, ID 선택자는 각각 언제 사용하는 것이 적절할까?

| 선택자 | 문법 | 사용 시기 | 예시 |
|--------|------|-----------|------|
| 태그 선택자 | `footer { }` | 해당 태그 **전체**에 공통 스타일을 적용할 때 | `body` 폰트, `footer` 정렬 |
| 클래스 선택자 | `.nav-links { }` | **여러 요소**에 재사용 가능한 스타일을 적용할 때 | `.til-item`, `.work-card` |
| ID 선택자 | `#til-form { }` | 페이지에서 **단 하나**뿐인 요소를 특정할 때 | `#image-modal`, `#til-list` |

실무에서는 **클래스 선택자를 가장 많이 사용**한다. ID 선택자는 우선순위(specificity)가 높아서 나중에 스타일을 덮어쓰기 어려우므로, 주로 JavaScript에서 요소를 선택할 때 사용하고 CSS에서는 클래스를 선호한다.

#### 3. CSS를 HTML 내에 직접 작성하는 것과 별도 파일로 분리하는 것의 차이는 무엇일까?

| 구분 | 인라인/내부 스타일 | 외부 파일 분리 |
|------|-------------------|---------------|
| 작성 위치 | `<style>` 태그 또는 `style` 속성 | `.css` 파일 → `<link>`로 연결 |
| 유지보수 | HTML과 CSS가 섞여 코드가 길어지고 찾기 어려움 | 역할별로 파일이 분리되어 관리가 쉬움 |
| 재사용 | 해당 HTML 파일에서만 사용 가능 | 여러 HTML 파일에서 같은 CSS를 공유 가능 |
| 캐싱 | 매번 HTML과 함께 다운로드 | 브라우저가 CSS 파일을 캐싱하여 재방문 시 로딩 속도 향상 |
| 관심사 분리 | 구조(HTML)와 표현(CSS)이 섞임 | 구조와 표현이 분리되어 협업에 유리 |

따라서 **외부 파일 분리가 권장**되며, 이 프로젝트에서도 `css/` 디렉터리에 별도 파일로 관리하고 있다.

### JavaScript

#### 1. `document.getElementById()`와 `document.querySelector()`의 차이는 무엇인가?

| 구분 | `getElementById()` | `querySelector()` |
|------|--------------------|--------------------|
| 선택 방식 | ID 이름만 전달 (`"til-form"`) | CSS 선택자 문법 사용 (`"#til-form"`, `".class"`, `"div > p"`) |
| 반환 | ID에 해당하는 단일 요소 | 선택자에 매칭되는 **첫 번째** 요소 |
| 유연성 | ID로만 검색 가능 | 클래스, 태그, 속성, 복합 선택자 등 다양하게 검색 가능 |
| 성능 | ID 검색에 특화되어 약간 더 빠름 | 범용적이라 약간 느리지만 체감 차이 없음 |

`querySelector()`가 더 유연하므로 현재 프로젝트에서는 `querySelector()`를 통일하여 사용했다.

#### 2. 자기소개 페이지에서 활용할 수 있는 이벤트에는 어떤 것들이 있을까?

| 이벤트 | 활용 예시 |
|--------|-----------|
| `click` | 갤러리 이미지 클릭 시 모달 열기, 다크 모드 토글 |
| `submit` | TIL 폼 등록 시 새 항목 추가 |
| `scroll` | 내비게이션 스크롤 시 현재 섹션 하이라이트 |
| `mouseover` / `mouseout` | 갤러리 이미지 호버 시 확대 효과 |
| `keydown` | ESC 키로 모달 닫기 |
| `input` | 폼 입력 시 실시간 글자 수 카운트 |
| `DOMContentLoaded` | 페이지 로드 완료 후 초기 설정 실행 |

이 프로젝트에서는 `submit`(TIL 등록)과 `click`(갤러리 모달)을 활용했다.

#### 3. innerHTML로 DOM을 조작할 때 주의해야 할 점은 무엇일까?

**XSS(Cross-Site Scripting) 공격에 취약하다.**

```javascript
// 위험한 예시
tilList.innerHTML += `<p>${userInput}</p>`;
```

사용자가 입력한 값에 `<script>alert('해킹')</script>` 같은 악성 코드가 포함되어 있으면, `innerHTML`은 이를 HTML로 해석하여 스크립트가 실행된다.

**안전한 대안: `createElement()` + `textContent`**
```javascript
const p = document.createElement("p");
p.textContent = userInput; // HTML이 아닌 순수 텍스트로 삽입
```

`textContent`는 입력값을 순수 텍스트로 처리하므로 스크립트가 실행되지 않는다. 이 프로젝트에서도 `innerHTML` 대신 `createElement()`와 `textContent`를 사용하여 안전하게 구현했다.

또한 `innerHTML`을 사용하면 해당 요소의 기존 자식 요소들이 모두 제거되고 다시 파싱되므로, 기존 요소에 바인딩된 이벤트 리스너가 사라지는 문제도 있다.

### 배포

#### 1. GitHub Pages로 배포한 페이지에서 TIL을 작성하면, 새로고침 시 데이터가 유지될까?

**유지되지 않는다.**

JavaScript로 DOM에 추가한 요소는 **브라우저 메모리(RAM)** 에만 존재한다. 새로고침하면 브라우저가 서버에서 HTML 파일을 다시 받아오고, JavaScript가 처음부터 다시 실행되기 때문에 동적으로 추가한 TIL 항목은 모두 사라진다.

GitHub Pages는 **정적 호스팅**이므로, 사용자가 입력한 데이터를 서버에 저장하는 기능이 없다.

#### 2. JavaScript로 추가한 데이터를 영구적으로 저장하려면 어떤 방법이 있을까?

| 방법 | 저장 위치 | 특징 |
|------|-----------|------|
| **`localStorage`** | 브라우저 | 간단하게 키-값 쌍으로 저장. 같은 브라우저에서만 유지됨. 용량 약 5MB |
| **`sessionStorage`** | 브라우저 | localStorage와 비슷하지만 탭을 닫으면 삭제됨 |
| **`IndexedDB`** | 브라우저 | 대용량 구조화 데이터 저장 가능. API가 복잡함 |
| **외부 DB + API 서버** | 서버 | Firebase, Supabase 등 BaaS 활용. 어떤 기기에서든 데이터 접근 가능 |
| **JSON 파일 + Git** | 저장소 | 데이터를 JSON 파일로 저장하고 커밋. 정적 사이트에서 활용 가능 |

이 프로젝트처럼 서버 없이 간단하게 구현하려면 **`localStorage`** 가 가장 현실적인 선택이다:

```javascript
// 저장
localStorage.setItem("tilData", JSON.stringify(tilArray));

// 불러오기
const saved = JSON.parse(localStorage.getItem("tilData"));
```
